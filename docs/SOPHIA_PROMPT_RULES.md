## 1. 공통 원칙

Sophia의 모든 스킬에 적용되는 기반 규칙.

### 1.1 데이터 정직성 원칙 (Data Honesty)

| 규칙                  | 설명                                                        |
| --------------------- | ----------------------------------------------------------- |
| 날조 금지             | 대화에 없는 수치를 만들지 않는다. 추정 시 "추정" 명시       |
| Y축 0 시작            | bar-chart, 비교 차트의 Y축은 반드시 0에서 시작              |
| 3D 차트 금지          | 모든 시각화는 2D 평면 — 3D는 데이터를 왜곡                  |
| 출처 표시             | 외부 데이터 인용 시 출처를 시각자료에 포함                   |
| 단위 명시             | 숫자에는 반드시 단위를 붙인다 (원, %, 명, 개월 등)          |

### 1.2 Negative Constraints (금지 사항)

```
프롬프트에 반드시 포함할 금지 항목:
- 마크다운 기호(#, *, -, ```)를 시각자료 텍스트에 포함하지 않는다
- 데이터를 날조하지 않는다 — 대화 맥락에 없으면 "데이터 없음" 표시
- 이모지를 차트/표 데이터에 넣지 않는다 (레이블에만 허용)
- 영문 전용 약어를 한국어 레이블에 혼용하지 않는다 (ROI → 투자수익률)
```

### 1.3 한국어 렌더링 힌트

```
프롬프트에 포함할 한국어 처리 지침:
- 한글 레이블: 최대 8자 권장 (차트 가독성)
- 긴 한글 레이블: 2줄 분할 허용 (예: "마케팅\n예산")
- 숫자+단위: "5,000만원" 형태 (쉼표 포함)
- 퍼센트: "35%" (한글 "퍼센트" 아닌 기호 사용)
```

### 1.4 브랜드 팔레트 주입

BizRoom 디자인 시스템의 색상을 Sophia의 모든 시각 출력에 일관 적용한다.

| 용도            | 색상 역할       | 값                          | 사용 맥락                        |
| --------------- | --------------- | --------------------------- | -------------------------------- |
| Primary         | 주요 강조       | Agent별 Color (§ Agent)     | 발언자의 차트 주 색상            |
| Positive        | 성장/긍정       | `#10B981` (Emerald)         | 매출 증가, 달성, 완료            |
| Negative        | 경고/부정       | `#EF4444` (Red)             | 예산 초과, 미달성, 리스크        |
| Neutral         | 배경/기본       | `#6B7280` (Gray)            | 보조 데이터, 비활성 항목         |
| Background      | 차트 배경       | `#1E293B` (Dark Slate)      | 빅스크린 어두운 배경 위 가독성   |

```
프롬프트 주입 예시:
"색상 규칙: 긍정적 수치는 #10B981, 부정적 수치는 #EF4444,
 중립은 #6B7280을 사용합니다. 배경은 어두운 톤(#1E293B)입니다."
```

## 2. Skill 1: Visual Generation — 빅스크린 시각자료

> 파일: `backend/src/agents/prompts/sophia.ts` → `SOPHIA_VISUAL_SYSTEM_PROMPT`
> 호출: `VoiceLiveOrchestrator.callSophiaVisualGPT()`
> 모델: Spark (fast), temperature: 0.2

### 2.1 프롬프트 구조 (4-Layer)

Maestiq PPT 서비스의 4계층 프롬프트 아키텍처를 적용한다.

```
Layer 1 — Base: Sophia 정체성 + 역할 정의
Layer 2 — Schema: type별 JSON 구조 정의 (현재 SOPHIA_VISUAL_SYSTEM_PROMPT)
Layer 3 — Rules: 데이터 추출 규칙 + 시각화 가이드라인
Layer 4 — Self-Verify: 생성 전 자체 검증 체크리스트
```

### 2.2 Type별 시각화 규칙

| Type           | 언제 사용                      | items 수 | 필수 데이터                             | 색상 규칙                           |
| -------------- | ------------------------------ | -------- | --------------------------------------- | ----------------------------------- |
| `comparison`   | A vs B 비교, 기존 vs 제안      | rows 3-6 | columns 2-3개, 모든 셀 채움            | 헤더: Primary, 행: 교차 음영        |
| `pie-chart`    | 비율/구성/점유율               | 3-6      | value 합계 = 100, 각 항목 color 필수   | 큰 비율 → 진한 색, 작은 비율 → 연한 |
| `bar-chart`    | 수치 크기 비교                 | 3-7      | value는 양의 정수, label 8자 이내      | 최대값 Primary, 나머지 Neutral      |
| `timeline`     | 일정/마일스톤/단계             | 3-5      | date + label + status(done/current/pending) | done: Positive, current: Primary, pending: Neutral |
| `checklist`    | 할 일/체크리스트               | 3-7      | text + checked(boolean)                | checked: Positive, unchecked: Neutral |
| `summary`      | 핵심 정리/요약                 | 3-5      | 각 포인트 30자 이내                    | 번호: Primary, 텍스트: White        |
| `architecture` | 시스템 구조/프로세스           | nodes 3-6 | node에 id/label/x/y, edge에 from/to   | node: Primary, edge: Neutral        |

### 2.3 데이터 추출 규칙

```
프롬프트에 포함할 데이터 추출 지침:

## 데이터 추출 우선순위
1. 대화에서 명시적으로 언급된 수치를 그대로 사용
2. 대화의 맥락에서 계산 가능한 수치를 도출 (예: "50%보다 높은" → 55-60% 추정)
3. 수치가 전혀 없으면 해당 type 대신 summary/checklist로 전환

## 추출 금지
- 에이전트가 말하지 않은 수치를 생성하지 않는다
- "약 100억" 같은 모호한 수치는 "~100억"으로 표기
- 두 에이전트가 다른 수치를 말한 경우 → comparison으로 양쪽 모두 표시
```

### 2.4 Self-Verify 체크리스트

Spark 모델 호출 시 프롬프트 마지막에 삽입:

```
## 생성 전 검증 (반드시 확인 후 출력)
1. ✓ type이 visual_hint.type과 일치하는가?
2. ✓ 모든 텍스트가 한국어인가?
3. ✓ value가 정수(소수점 없음)인가?
4. ✓ items 수가 3-7개 범위인가?
5. ✓ 대화에 없는 수치를 만들지 않았는가?
6. ✓ 마크다운 기호(#, *, -)가 텍스트에 포함되지 않았는가?
7. ✓ color가 hex 코드이고 pie-chart에만 사용했는가?
8. ✓ 레이블이 8자 이내인가?
```

### 2.5 시각 메타데이터 (확장 필드)

현재 BigScreenRenderData에 선택적 메타데이터를 추가하여 렌더링 품질을 향상한다.

```typescript
// 향후 BigScreenRenderData에 추가 가능한 선택 필드
interface VisualMetadata {
  visualWeight: "visual_heavy" | "text_heavy" | "balanced";
  colorTemperature: "warm" | "cool" | "neutral";
  // warm: 성장/긍정적 주제 (매출 증가, 달성)
  // cool: 분석/기술적 주제 (아키텍처, 비용 분석)
  // neutral: 일반 정보 (체크리스트, 요약)
  dataVisualizationHint: string;
  // 예: "bar_chart_with_growth_trend", "process_flow_4_steps"
}
```

| colorTemperature | 대화 맥락                        | 적용 색 톤                    |
| ---------------- | -------------------------------- | ----------------------------- |
| `warm`           | 매출 증가, 성장, 긍정적 전망     | Emerald, Amber 계열           |
| `cool`           | 비용 분석, 기술 검토, 리스크     | Blue, Slate 계열              |
| `neutral`        | 일반 정리, 체크리스트, 요약      | Gray, White 계열              |

## 3. Skill 2: Image Prompt Generation — 이미지 프롬프트

> 적용 시점: PPT 산출물 슬라이드에 삽입할 일러스트레이션 생성 시
> 모델: DALL-E 3 또는 Azure AI Image Generation
> 상태: 현재 Non-Goal (§1.4), CDO Jonas 크리에이티브 비주얼로 향후 확장

### 3.1 이미지 프롬프트 작성 규칙

PPT 슬라이드나 빅스크린 배경에 사용할 이미지 프롬프트의 작성 기준.

```
## 필수 스타일 프리픽스
"Professional flat design illustration, modern corporate style,
 clean minimal, 16:9 aspect ratio"

## 필수 네거티브 서픽스
"NO text, NO labels, NO watermarks, NO logos,
 pure visual illustration only"

## 톤 매핑 (슬라이드 주제 → 색상 톤)
- 성장/매출/기회 → "warm corporate tones, green and amber accents"
- 분석/기술/데이터 → "cool blue tones, tech-forward palette"
- 리스크/경고/법적 → "muted tones with cautionary amber highlights"
- 일반/중립 → "modern corporate blue and gray palette"
```

### 3.2 주제별 이미지 프롬프트 템플릿

| 슬라이드 주제       | 이미지 프롬프트 키워드                                                                |
| ------------------- | ------------------------------------------------------------------------------------- |
| 시장 분석           | rising market growth chart, upward arrows, data visualization elements                |
| 재무 비교           | balance scale with coins, financial comparison visualization, dual-sided infographic  |
| 마케팅 전략         | target audience segments, campaign funnel, digital marketing channels                 |
| 기술 아키텍처       | abstract system diagram, connected nodes, cloud infrastructure visualization          |
| 팀/조직             | collaborative team silhouettes, organizational hierarchy, connected people            |
| 타임라인/로드맵     | horizontal timeline with milestones, progressive stages, journey path                 |
| 액션아이템          | checklist with checkmarks, task management board, organized workflow                  |
| 의사결정            | crossroads visualization, decision tree branches, strategic choice                    |

### 3.3 이미지 프롬프트 생성 프롬프트 (Meta-Prompt)

Sophia가 슬라이드 내용을 기반으로 imagePrompt를 자동 생성할 때 사용하는 지침:

```
당신은 비즈니스 프레젠테이션 이미지 프롬프트 전문가입니다.
슬라이드 제목과 내용을 바탕으로 영문 이미지 프롬프트를 작성합니다.

## 작성 규칙
1. 영문으로 작성 (이미지 생성 AI는 영문 최적화)
2. 슬라이드 핵심 개념을 시각적 은유로 표현
3. 구체적 시각 요소를 3-5개 포함
4. 추상적 묘사("좋은 느낌") 금지 — 구체적 오브젝트로 표현
5. 500자 이내

## 필수 포함
- 스타일: "Professional flat design illustration, modern corporate style"
- 비율: "16:9 aspect ratio"
- 금지: "NO text, NO labels, NO watermarks"

## 금지
- 실제 인물 묘사 (얼굴, 인종 특정)
- 특정 브랜드 로고나 제품
- 텍스트/글자/라벨이 포함된 이미지
- 사진 스타일 (항상 일러스트레이션)

## 예시
입력: "A안 vs B안 비용 비교 — A안 300만원, B안 180만원"
출력: "Professional flat illustration of two parallel paths diverging
from a single point, left path wider with more resource icons and amber
tones representing higher cost, right path leaner with green efficiency
symbols, abstract financial comparison visualization, modern corporate
blue and white palette, clean minimal style, 16:9 aspect ratio,
NO text or labels"
```

### 3.4 브랜드 일관성 체크

| 검증 항목                 | 기준                                                    |
| ------------------------- | ------------------------------------------------------- |
| 스타일 일관성             | 모든 슬라이드가 동일한 flat design 스타일                |
| 색상 톤 일관성            | warm/cool/neutral이 슬라이드 주제와 일치                |
| 텍스트 미포함             | 생성된 이미지에 글자가 절대 없음                        |
| 비율                      | 16:9 (빅스크린/PPT 호환)                                |
| 문화적 중립               | 특정 문화/인종/종교를 특정하지 않음                     |

## 4. Skill 3: PPT Artifact — 회의 산출물 PPT

> 파일: `backend/src/services/ArtifactGenerator.ts` → `generatePPT()`
> 호출: 회의 종료 시 Sophia가 자동 생성
> 라이브러리: pptxgenjs

### 4.1 결론형 제목 규칙

```
PPT 슬라이드 제목은 "결론형"으로 작성한다.
설명형 제목이 아닌, 핵심 결론을 담은 제목.

❌ 잘못된 예:
- "마케팅 전략 논의"
- "예산 현황"
- "다음 단계"

✅ 올바른 예:
- "SNS 마케팅 중심 3개월 파일럿, 예산 300만원 확정"
- "현금흐름 여유 2.3억, 신규 투자 가능 구간 확인"
- "4월 론칭 목표, 3단계 실행 로드맵 합의"

## 규칙
- 숫자가 있으면 반드시 제목에 포함
- 결정 사항이면 결정 내용을 제목으로
- 토론 정리이면 핵심 합의를 제목으로
- 15자 이내 권장, 최대 25자
```

### 4.2 서사 구조 (Story Arc)

회의 산출물 PPT 전체를 4막 구조로 배치한다.

```
┌─────────────────────────────────────────────────────────────┐
│  Act 1: 배경 (Context)                                      │
│  - 오늘 회의의 안건과 배경                                  │
│  - 왜 이 논의가 필요했는가                                  │
│  - 시장/상황 데이터 (웹 검색 결과 포함)                     │
│  ↓ highlight 슬라이드: 핵심 문제 한 줄                      │
├─────────────────────────────────────────────────────────────┤
│  Act 2: 논의 (Discussion)                                    │
│  - 각 안건별 에이전트들의 핵심 의견                         │
│  - 비교/대안 (comparison, bar-chart 활용)                   │
│  - 반박과 보완 포인트                                        │
│  ↓ highlight 슬라이드: 핵심 인사이트                        │
├─────────────────────────────────────────────────────────────┤
│  Act 3: 결정 (Decision)                                      │
│  - Chairman 결정 사항                                        │
│  - 결정의 근거와 조건                                        │
│  - 시각자료 임베드 (회의 중 빅스크린에 띄운 것)             │
│  ↓ highlight 슬라이드: 최종 결정 한 줄                      │
├─────────────────────────────────────────────────────────────┤
│  Act 4: 실행 (Action)                                        │
│  - 액션아이템 테이블                                         │
│  - 타임라인/마일스톤                                         │
│  - 다음 회의 안건 예고                                       │
│  ↓ 마무리 슬라이드: 감사 + 다음 단계                        │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 레이아웃 선택 로직

회의록 데이터 → PPT 슬라이드 레이아웃 자동 선택:

| 데이터 특성                           | 레이아웃           | 적용 예시                            |
| ------------------------------------- | ------------------ | ------------------------------------ |
| 핵심 3-5개 불릿                       | `bullets`          | 안건 요약, 결정 사항                 |
| A vs B 비교 구조                      | `two_column`       | A안/B안, 기존/제안, 비용/효과        |
| 핵심 한 줄 메시지 (임팩트)            | `highlight`        | 최종 결정, 핵심 인사이트, 비전       |
| 수치 데이터 3개+ 카테고리             | `chart`            | 예산 비교, 매출 전망, 시장 규모      |
| 일정/단계 3-5단계                     | `timeline`         | 실행 로드맵, 마일스톤, 프로젝트 일정 |
| 할 일/체크 항목                       | `checklist`        | 액션아이템, 검증 리스트              |

### 4.4 불릿 작성 규칙

```
## PPT 불릿 작성법
- 한 불릿 = 한 문장 (30자 이내, 명사+동사 중심)
- 구체적 숫자는 항상 포함: "매출 15% 증가" ✓  "매출 증가" ✗
- 6개 이상이면 2개 슬라이드로 분할
- 에이전트 이름이 불릿에 들어가지 않는다 (발표자 노트에서 언급)
- 약어는 처음 사용 시 풀어쓴다: "CAC(고객획득비용) 50만원"
```

### 4.5 발표자 노트 (Speaker Notes)

```
## 발표자 노트 작성 규칙
- 각 슬라이드에 speakerNotes를 2-3문장으로 작성
- 자연스러운 구어체 (발표 스크립트 느낌)
- 슬라이드의 불릿을 그대로 읽지 말고, 맥락과 의미를 설명
- 에이전트 이름 인용 가능: "CFO Amelia가 강조한 바와 같이..."
- 결정의 배경 설명: "이 결정은 현금흐름 여유를 고려한 결과입니다"

## 예시
슬라이드: "SNS 마케팅 파일럿 300만원 확정"
Speaker Notes: "이 안건에서는 CMO Yusef가 제안한 SNS 중심 마케팅 전략에 대해
논의했습니다. CFO Amelia의 예산 검토를 거쳐 300만원 규모의 3개월
파일럿으로 확정되었으며, ROI 120% 이상 달성 시 확대 집행하기로
합의했습니다."
```

### 4.6 분할 규칙

| 조건                              | 분할 기준                                              |
| --------------------------------- | ------------------------------------------------------ |
| 안건 논의 내용이 500자 이상       | 핵심 요약 슬라이드 + 상세 슬라이드로 분할              |
| 액션아이템이 6개 이상             | 담당자별 또는 우선순위별로 2개 슬라이드 분할            |
| 에이전트 의견이 3명 이상 대립     | 의견 비교 슬라이드 + 결론 슬라이드로 분할              |
| 전체 슬라이드 수                  | 최소 5장, 최대 15장 (표지/마무리 제외)                 |

### 4.7 시각자료 임베드

```
회의 중 빅스크린에 표시된 시각자료(sophia.visualHistory)를
PPT 슬라이드에 이미지로 임베드한다.

## 임베드 규칙
- 해당 안건 슬라이드에 관련 시각자료를 배치
- 시각자료가 2개 이상이면 가장 마지막(최종) 버전만 사용
- 시각자료 캡션: "[회의 중 생성] {title}" 형식
- 이미지 크기: 슬라이드 폭의 60-80%
```

## 5. Skill 4: Meeting Minutes — 회의록

> 파일: `backend/src/agents/prompts/sophia.ts` → `SOPHIA_MINUTES_SYSTEM_PROMPT`
> 모델: GPT-5.4 (premium), temperature: 0.4

### 5.1 Chain-of-Thought 강화

현재 CoT 구조에 서사적 연결(transitionLogic)을 추가한다.

```
## 회의록 생성 프로세스 (단계별 사고)

1단계 — 전체 흐름 파악:
  "이 회의에서 다룬 핵심 안건은 무엇이었나?"
  "각 안건의 논의 흐름은 어떠했나? (제안 → 반박 → 수정 → 결정)"

2단계 — 안건별 정리:
  "이 안건에서 어떤 결정이 내려졌나?"
  "결정되지 않은 미결 사항은 무엇인가?"
  "이 안건과 다음 안건은 어떻게 연결되는가?" ← 서사적 연결

3단계 — 액션아이템 추출:
  "누가, 무엇을, 언제까지 해야 하는가?"
  "암묵적으로 합의된 액션아이템은 없는가?" ← 명시적 요청 외 추출

4단계 — 최종 검증:
  "모든 결정 사항이 회의록에 반영되었는가?"
  "누락된 에이전트 발언은 없는가?"
  "수치가 정확한가? (대화 원문과 대조)"
```

### 5.2 회의록 품질 기준

| 검증 항목                   | 기준                                                    |
| --------------------------- | ------------------------------------------------------- |
| 결정 사항 완전성            | Chairman이 결정한 모든 사항이 포함                      |
| 액션아이템 매핑             | 모든 액션에 담당자(assignee)가 지정                     |
| 수치 정확성                 | 대화에서 언급된 수치와 회의록 수치가 일치               |
| 에이전트 발언 균형          | 발언한 에이전트 모두의 핵심 포인트가 요약에 반영         |
| 미결 사항 구분              | 결정된 것과 미결 사항이 명확히 분리                     |
| 요약 vs 복사                | 원문 복사가 아닌 핵심 요약으로 재구성                   |

### 5.3 서사적 연결 (Transition Logic)

```
각 안건의 agendas 항목에 transitionLogic 필드를 추가하여
안건 간 논리적 흐름을 연결한다.

예시:
agendas: [
  {
    title: "SNS 마케팅 파일럿 예산",
    summary: "...",
    transitionLogic: "이 예산 결정을 바탕으로 구체적 실행 일정을 논의"
  },
  {
    title: "마케팅 실행 타임라인",
    summary: "...",
    transitionLogic: "타임라인 확정 후 기술 지원 범위를 검토"
  }
]
```

## 6. Document Transformation — 문서 변환 체계

> Sophia는 동일한 회의 버퍼(buffer)에서 목적에 따라 다른 형태의 문서를 생성한다.
> 아래는 모드 선택 로직과 각 출력 형태의 관계를 정의한다.

### 6.1 출력 모드 전체 맵

```
회의 종료 시 Sophia.buffer (전체 대화 기록)
    │
    ├─ [자동] Skill 4: Meeting Minutes → 구조화된 회의록 JSON
    │     └─ [자동] Skill 3: Full PPT (5-15장) → .pptx
    │
    └─ [사용자 요청 시] Document Transformation
          │
          ├─ "보고서로 정리해줘"
          │     └─ Skill 5: Executive Report → 경영 보고서
          │
          ├─ "핵심만 장표로", "2-3장으로 압축"
          │     └─ Skill 6: Slide Deck (2-5장) → 압축 .pptx
          │
          └─ "사업계획서로 만들어줘", "IR 자료"
                └─ Skill 7: Business Plan → 사업계획서 .pptx/.docx
```

### 6.2 모드 선택 로직 (Trigger Keywords)

| 모드               | 트리거 키워드                                                      | 자동/수동 | 출력 형식    |
| ------------------ | ------------------------------------------------------------------ | --------- | ------------ |
| Meeting Minutes    | 회의 종료 시 자동                                                  | 자동      | JSON → PPT   |
| Full PPT           | 회의 종료 시 자동 (회의록 기반)                                    | 자동      | .pptx 5-15장 |
| Executive Report   | "보고서", "리포트", "보고용", "경영진 보고", "상사 보고"           | 수동      | .docx / .pptx |
| Slide Deck         | "핵심만", "장표", "압축", "요약본", "한 눈에", "N장으로"          | 수동      | .pptx 2-5장  |
| Business Plan      | "사업계획서", "IR 자료", "투자 유치", "피칭덱", "BM 정리"          | 수동      | .pptx 8-15장 |

### 6.3 공유 데이터 소스

모든 문서 변환 모드는 동일한 소스에서 데이터를 추출한다:

```typescript
interface DocumentTransformSource {
  // 1차 소스: Sophia 내부 버퍼
  buffer: SophiaBufferEntry[];        // 전체 대화 기록
  decisions: string[];                // Chairman 결정 사항
  actionItems: ActionItemDraft[];     // 액션아이템
  visualHistory: VisualArtifact[];    // 빅스크린에 띄운 시각자료

  // 2차 소스: 회의 메타데이터
  meetingInfo: {
    title: string;
    date: string;
    agenda: string[];
    participants: string[];
  };

  // 3차 소스 (선택): Brand Memory Set
  brandContext?: BrandMemorySet;      // 회사명, 산업, 스타일 가이드
}
```

### 6.4 분량 자동 조절 원칙

입력 분량(대화 길이)에 따라 출력 분량을 자동 조절한다:

| 대화 버퍼 크기            | Minutes    | Full PPT   | Report     | Slide Deck | Biz Plan    |
| ------------------------- | ---------- | ---------- | ---------- | ---------- | ----------- |
| 소형 (10분, ~20턴)        | 1페이지    | 5-7장      | 1-2페이지  | 2장        | 8장 (골격)  |
| 중형 (30분, ~50턴)        | 2-3페이지  | 8-12장     | 3-4페이지  | 3장        | 10-12장     |
| 대형 (60분+, ~100턴)      | 4-5페이지  | 12-15장    | 5-7페이지  | 4-5장      | 12-15장     |

## 7. Skill 5: Executive Report — 경영 보고서

> 트리거: "보고서로 정리해줘", "경영진 보고용으로 만들어줘"
> 모델: GPT-5.4 (premium), temperature: 0.4
> 출력: 문서 형태 (Word/PDF 또는 슬라이드)

### 7.1 보고서 구조

회의 내용을 상사/경영진에게 보고하는 공식 문서 형태로 변환한다.

```
┌─────────────────────────────────────────────────────────────┐
│  ① 표지                                                     │
│     제목 (결론형), 일시, 작성자, 참석자                      │
├─────────────────────────────────────────────────────────────┤
│  ② Executive Summary (핵심 3-5줄)                           │
│     "이번 회의에서 3가지 안건을 논의하여                     │
│      2건 확정, 1건 추가 검토로 결정."                        │
│     → 읽는 사람이 이것만 읽어도 전체 파악 가능               │
├─────────────────────────────────────────────────────────────┤
│  ③ 핵심 결정 사항 (Decision Summary)                         │
│     결정 1: [내용] — 조건: [조건] — 근거: [한 줄]            │
│     결정 2: ...                                               │
│     미결: [내용] — 사유: [한 줄] — 기한: [날짜]              │
├─────────────────────────────────────────────────────────────┤
│  ④ 안건별 논의 요약                                          │
│     안건 1: [결론형 제목]                                     │
│       - 배경: ...                                             │
│       - 주요 논점: ...                                        │
│       - 결론: ...                                             │
│     안건 2: ...                                               │
├─────────────────────────────────────────────────────────────┤
│  ⑤ 액션아이템                                                │
│     표: 담당자 | 내용 | 기한 | 상태                           │
├─────────────────────────────────────────────────────────────┤
│  ⑥ 부록 (선택)                                               │
│     - 회의 중 생성된 시각자료                                 │
│     - 참고 데이터 (웹 검색 결과 등)                           │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 보고서 작성 프롬프트 규칙

```
당신은 BizRoom.ai의 경영 보고서 작성 어시스턴트입니다.
회의 기록을 바탕으로 상사/경영진에게 보고하는 공식 문서를 작성합니다.

## 문체 규칙
- 격식체 사용 ("~입니다", "~했습니다")
- 결론 선행: 각 섹션의 첫 문장이 해당 섹션의 결론
- 숫자 중심: 모든 정량적 내용에 수치 포함
- 간결 우선: 1문장으로 쓸 수 있으면 2문장으로 쓰지 않는다

## Executive Summary 작성법
- 3-5줄로 회의 전체를 압축
- 패턴: "[N개 안건 논의] → [M건 확정] → [핵심 결정 한 줄]"
- 읽는 사람이 이것만 읽고도 "무엇이 결정되었는가"를 파악 가능해야 한다

## 결정 사항 표기법
- 확정: ✅ [결정 내용] — 조건: [있으면] — 근거: [한 줄]
- 미결: ⏳ [미결 내용] — 사유: [한 줄] — 재논의 기한: [날짜]
- 기각: ❌ [기각 내용] — 사유: [한 줄]

## 에이전트 표기
- 보고서에서 에이전트를 직접 언급하지 않는다
- "재무 검토 결과..." (Amelia가 아닌 기능 중심으로)
- "마케팅 관점에서..." (Yusef가 아닌 관점 중심으로)
- 사용자의 결정은 "의장(Chairman)" 또는 실명으로 표기

## 데이터 정직성
- 회의에서 언급된 수치만 사용
- 추정치는 "추정" 명시
- 외부 검색 데이터는 출처 표기
```

### 7.3 보고서 Self-Verify

```
## 보고서 생성 전 검증
1. ✓ Executive Summary만 읽어도 전체 파악이 되는가?
2. ✓ 모든 결정 사항이 Decision Summary에 포함되었는가?
3. ✓ 미결 사항에 재논의 기한이 명시되었는가?
4. ✓ 액션아이템에 담당자와 기한이 모두 있는가?
5. ✓ 에이전트 이름(Hudson, Amelia 등)이 본문에 노출되지 않았는가?
6. ✓ 모든 수치에 단위가 붙어있는가?
```

## 8. Skill 6: Slide Deck Compression — 핵심 장표 2-5장

> 트리거: "핵심만 장표로", "2-3장으로 정리", "한 눈에 보게 해줘"
> 모델: GPT-5.4 (premium), temperature: 0.3
> 출력: .pptx 2-5장 (극한 압축)

### 8.1 설계 철학

```
"1시간 회의를 엘리베이터에서 30초 만에 전달할 수 있는가?"

Full PPT (5-15장)이 회의의 '기록'이라면,
Slide Deck (2-5장)은 회의의 '결론'이다.

원칙:
- 1 슬라이드 = 1 메시지 (Only One Takeaway)
- 숫자 > 문장 > 단어 (정보 밀도 극대화)
- highlight 레이아웃 적극 활용 (임팩트)
- 읽지 않아도 '보면' 이해되는 수준
```

### 8.2 슬라이드 구성 (대화 분량별)

#### 2장 구성 (10분 미만 짧은 회의)

| 장   | 내용                                      | 레이아웃      |
| ---- | ----------------------------------------- | ------------- |
| 1    | 결정 사항 + 핵심 숫자 1개                 | `highlight`   |
| 2    | 액션아이템 (누가, 무엇을, 언제)           | `checklist`   |

#### 3장 구성 (일반 회의, 가장 빈번)

| 장   | 내용                                      | 레이아웃      |
| ---- | ----------------------------------------- | ------------- |
| 1    | 핵심 결론 한 줄 + keyFigure               | `highlight`   |
| 2    | 결정 사항 2-3개 (불릿)                    | `bullets`     |
| 3    | 액션아이템 + 다음 단계                    | `checklist`   |

#### 5장 구성 (긴 회의, 안건 다수)

| 장   | 내용                                      | 레이아웃      |
| ---- | ----------------------------------------- | ------------- |
| 1    | 회의 핵심 한 줄 + keyFigure               | `highlight`   |
| 2    | 주요 결정 사항                            | `bullets`     |
| 3    | 핵심 데이터 (차트/비교)                   | `chart` / `two_column` |
| 4    | 액션아이템                                | `checklist`   |
| 5    | 타임라인 / 다음 단계                      | `timeline`    |

### 8.3 압축 프롬프트 규칙

```
당신은 BizRoom.ai의 장표 압축 전문가입니다.
회의 기록을 {N}장의 핵심 장표로 압축합니다.

## 압축 원칙
1. "So What?" 테스트: 각 슬라이드를 보고 "그래서 뭐?"라는 질문에
   바로 답할 수 있어야 한다. 답할 수 없으면 그 슬라이드는 불필요.
2. 결정 > 논의 > 배경: 압축 시 배경 설명부터 잘라낸다.
   결정된 것만 남기고, 왜 그렇게 결정했는지는 1문장으로.
3. 숫자가 말한다: 텍스트 3줄보다 숫자 1개가 낫다.
   keyFigure가 있으면 슬라이드 중앙에 크게 배치.

## 불릿 작성법 (압축 모드)
- 일반 PPT 불릿: 30자 이내
- 압축 장표 불릿: 15자 이내 (명사구 + 숫자)
- 예시:
  일반: "SNS 마케팅 중심 3개월 파일럿으로 300만원 집행 확정"
  압축: "SNS 파일럿 300만/3개월 확정"

## keyFigure 사용법
- 슬라이드에서 가장 임팩트 있는 숫자 1개를 keyFigure로 추출
- value: "2.3억", "300만원", "35%", "3개월" 등
- label: 한 줄 설명 (10자 이내)
- highlight 레이아웃에서 중앙 배치, 큰 폰트

## 생략 규칙 (뭘 버리는가)
- 에이전트 간 토론 과정 → 생략 (결론만)
- 배경/맥락 설명 → 생략 (결정 사항에 1문장으로 흡수)
- 반박/보완 세부 내용 → 생략 (최종 합의만)
- 면책 조항 → 생략 (보고서/Full PPT에만)
- 인사/마무리 → 생략
```

### 9.4 데이터 부족 처리 — "추가 필요" 마커

```
사업계획서의 모든 섹션이 회의에서 논의되지는 않는다.
대화에서 추출할 수 없는 내용은 날조하지 않고,
"추가 필요" 마커를 달아 사용자에게 보완을 요청한다.

## 마커 표기법
- 슬라이드 불릿에: "[추가 필요] 정확한 TAM/SAM 데이터"
- speakerNotes에: "이 부분은 회의에서 구체적으로 논의되지 않았습니다.
  실제 시장 데이터를 보완해주세요."
- 수치 데이터: "[추정] ~50억" 또는 "[미정] 추가 조사 필요"

## 절대 하지 않는 것
- 시장 규모를 추측으로 채우지 않는다
- 매출 전망을 근거 없이 만들지 않는다
- 경쟁사를 대화에 없는 내용으로 비교하지 않는다
- "추가 필요" 마커가 3개 이상이면 해당 슬라이드 생성 여부를
  사용자에게 먼저 확인한다
```

### 9.5 사업계획서 프롬프트

```
당신은 BizRoom.ai의 사업계획서 작성 어시스턴트입니다.
회의 기록을 바탕으로 투자자/심사위원 대상 피칭덱을 작성합니다.

## 작성 원칙
1. 결론형 제목: "사업 개요" → "{회사명}, {핵심 가치 한 줄}"
2. 숫자 우선: 정량적 근거가 있으면 반드시 슬라이드에 포함
3. Story Arc: 문제→솔루션→증거→비전 4막 구조
4. 데이터 정직성: 대화에 없는 내용은 [추가 필요] 마커
5. 투자자 관점: "왜 이 팀이, 왜 지금, 왜 이 시장"에 답할 것

## 심사위원/투자자 주목 포인트 (judgeAttentionPoint)
각 슬라이드에 "이 슬라이드에서 심사위원이 주목할 포인트" 한 줄을 메타데이터로 추가:
- 예: "시장 규모의 구체적 근거가 있는가?"
- 예: "경쟁사 대비 명확한 차별점이 있는가?"
- 예: "팀이 이 문제를 풀 적임자인가?"

## 이미지 프롬프트
모든 슬라이드에 imagePrompt를 생성한다 (Skill 2 규칙 적용).
톤: 성장/기회 → warm, 분석/데이터 → cool, 비전 → warm

## 출력 형식
각 슬라이드: {
  title: 결론형 제목,
  layout: bullets|two_column|highlight|chart|timeline|checklist,
  content: { ... },
  keyFigure?: { value, label },
  speakerNotes: 2-3문장,
  imagePrompt: 영문 500자,
  judgeAttentionPoint: 한 줄,
  sourceField: 회의 안건 참조
}
```

### 9.6 사업계획서 Self-Verify

```
## 사업계획서 생성 전 검증
1. ✓ Story Arc 4막 구조가 완성되었는가?
2. ✓ 문제 정의 → 솔루션 → 증거 → 비전 흐름이 자연스러운가?
3. ✓ 대화에 없는 내용을 날조하지 않았는가?
4. ✓ [추가 필요] 마커가 적절히 표시되었는가?
5. ✓ keyFigure가 최소 3개 슬라이드에 포함되었는가?
6. ✓ 슬라이드 제목이 모두 결론형인가?
7. ✓ 슬라이드 수가 8-15장 범위인가?
8. ✓ judgeAttentionPoint가 모든 슬라이드에 있는가?
9. ✓ 모든 슬라이드에 imagePrompt가 있는가?
10. ✓ 회사명이 표지와 본문에 정확히 사용되었는가?
```

## 11. Sophia as Analysis Skill Hub

> 에이전트가 커스텀 가능하므로, 분석 프레임워크는 Sophia에 집중. 어떤 에이전트든 Sophia에게 스킬을 요청할 수 있다.

### 11.1 설계 원칙

| 원칙                    | 설명                                                      |
| ----------------------- | --------------------------------------------------------- |
| 에이전트 독립성         | 기본/커스텀 에이전트 모두 동일한 스킬 세트 접근 가능      |
| 단일 관리 포인트        | 스킬 추가/수정 = Sophia만 업데이트                        |
| 에이전트 토큰 최소화    | 에이전트 프롬프트에는 스킬 목록 1줄만 추가                |
| visual_hint 연결        | 모든 분석 결과는 BigScreen 시각화로 자연스럽게 이어짐     |

### 11.2 에이전트 프롬프트 추가 (공통, ~1줄)

모든 에이전트(기본 + 커스텀)의 common prompt에 다음 1줄을 추가:

```
분석이 필요할 때 visual_hint로 Sophia 스킬을 요청하세요: SWOT, ROI분석, BEP, TAM/SAM/SOM, 우선순위매트릭스, 경쟁포지셔닝, UX휴리스틱, 법적리스크체크
```

### 11.3 분석 스킬 카탈로그

#### AS-1: SWOT 분석
- **트리거**: 실행 전략, 시장 진입, 신규 사업 논의
- **입력**: visual_hint `{type: "comparison", title: "SWOT 분석"}`
- **Sophia 처리**: 대화 맥락에서 강점·약점·기회·위협 4축 추출 → comparison 렌더링
- **출력**: 2×2 comparison 테이블 (columns: ["", "내부", "외부"], rows: [["긍정", S, O], ["부정", W, T]])

#### AS-2: ROI 분석
- **트리거**: 지출 제안, 투자 논의
- **입력**: visual_hint `{type: "bar-chart", title: "ROI 분석"}`
- **Sophia 처리**: 비용·수익·회수기간 추출 → bar-chart 렌더링
- **출력**: bar-chart (항목: 투자비용, 예상수익, 순이익)

#### AS-3: BEP (손익분기점)
- **트리거**: 신규 수익 모델, 가격 설정
- **입력**: visual_hint `{type: "timeline", title: "손익분기점 분석"}`
- **Sophia 처리**: 고정비/(단가-변동비) 계산 → timeline 렌더링
- **출력**: timeline (월별 누적 손익, BEP 시점 "current" 마커)

#### AS-4: TAM/SAM/SOM
- **트리거**: 시장 규모, 타겟 시장 논의
- **입력**: visual_hint `{type: "pie-chart", title: "시장 규모 분석"}`
- **Sophia 처리**: 전체 시장(TAM) → 유효 시장(SAM) → 목표 시장(SOM) 추출 → pie-chart
- **출력**: pie-chart (3개 항목, 중첩 비율)

#### AS-5: 우선순위 매트릭스
- **트리거**: 안건 다수, 우선순위 결정 필요
- **입력**: visual_hint `{type: "comparison", title: "우선순위 매트릭스"}`
- **Sophia 처리**: 중요도×긴급도로 즉시/계획/위임/보류 분류 → comparison
- **출력**: comparison (columns: ["안건", "중요도", "긴급도", "분류"])

#### AS-6: 경쟁사 포지셔닝
- **트리거**: 경쟁 분석, 차별화 전략
- **입력**: visual_hint `{type: "comparison", title: "경쟁사 포지셔닝"}`
- **Sophia 처리**: 경쟁사별 특성 비교 → comparison 렌더링
- **출력**: comparison (columns: ["특성", "우리", "경쟁사A", "경쟁사B"])

#### AS-7: UX 휴리스틱 평가
- **트리거**: UI/UX 검토, 사용성 논의
- **입력**: visual_hint `{type: "checklist", title: "UX 휴리스틱 평가"}`
- **Sophia 처리**: 피드백·오류방지·일관성 3축 평가 → checklist
- **출력**: checklist (항목별 통과/미통과)

#### AS-8: 법적 리스크 체크
- **트리거**: 규제, 개인정보, 법적 검토 논의
- **입력**: visual_hint `{type: "checklist", title: "법적 리스크 체크"}`
- **Sophia 처리**: 주요 법적 리스크 항목 추출 → checklist
- **출력**: checklist (리스크 항목별 대응 여부)

### 11.4 스킬 → 시각화 매핑 요약

| 스킬                 | visual_hint.type | 주 활용 에이전트      |
| -------------------- | ---------------- | --------------------- |

| ROI 분석             | bar-chart        | CFO, 투자 논의        |
| BEP                  | timeline         | CFO, 수익 모델        |
| TAM/SAM/SOM          | pie-chart        | CMO, 시장 분석        |
| 우선순위 매트릭스    | comparison       | COO, 안건 관리        |
| 경쟁사 포지셔닝      | comparison       | CMO, 차별화           |
| UX 휴리스틱          | checklist        | CDO, 사용성           |
| 법적 리스크          | checklist        | CLO, 규제             |
