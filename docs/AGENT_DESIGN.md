# BizRoom.ai Agent Design Document

> AI C-Suite 임원진과 실시간 회의하는 가상 사무실 -- Microsoft Leadership에서 영감을 받은 멀티에이전트 그룹 대화 환경 설계 문서

---

## 1. 프로젝트 개요

### 1.1 비전

BizRoom.ai는 사용자가 **AI C-Suite 임원진과 실시간으로 회의**하는 가상 사무실이다. 각 에이전트는 **실제 Microsoft C-Suite 임원의 사고방식과 가치관에서 영감을 받아** 고유한 전문성, 성격, 의사결정 스타일을 가지며, 서로 보완하고 반박하며 현실적인 경영 회의를 시뮬레이션한다.

### 1.2 핵심 차별화: Microsoft Leadership Inspired Agents

BizRoom.ai의 에이전트는 단순한 역할 기반 AI가 아니다. 각 에이전트는 **실존하는 Microsoft 경영진**의 리더십 철학, 의사결정 패턴, 가치관에서 영감을 받아 설계되었다.

| 설계 원칙                             | 설명                                                              |
| ------------------------------------- | ----------------------------------------------------------------- |
| **영감 기반 페르소나**                | 실제 경영진의 공개 발언, 저서, 인터뷰에서 추출한 가치관 반영      |
| **깊이 있는 의사결정 패턴**           | 단순 역할이 아닌, 해당 인물의 사고 프레임워크를 반영한 의사결정   |
| **Microsoft 해커톤 차별화 포인트**    | Microsoft 문화와 리더십에 대한 존경을 제품에 녹여낸 전략          |
| **현실적 경영 시뮬레이션**            | 실제 C-Suite 레벨의 의사결정 깊이와 갈등 역학 재현               |

### 1.3 핵심 기술 스택

| 기술                              | 용도                                                    |
| --------------------------------- | ------------------------------------------------------- |
| Microsoft Semantic Kernel         | 에이전트 오케스트레이션 프레임워크                       |
| Azure OpenAI Service              | LLM 추론 엔진 (GPT-4o)                                 |
| Azure AI Foundry Model Router     | 에이전트별 최적 모델 자동 라우팅 (GPT-4o / GPT-4o-mini) |
| GPT Realtime API                  | 실시간 음성 대화, 인터럽트, 턴테이킹 (WebRTC)           |
| Microsoft Graph API               | OneDrive 파일 임베드, Calendar, Planner 연동            |
| DialogLab 개념                    | 멀티에이전트 그룹 대화 구조 설계                        |
| Azure MCP                         | 외부 도구 연결 (Calendar, Notion, GitHub 등)            |
| SignalR / WebSocket               | 실시간 대화 스트리밍                                    |

### 1.4 DialogLab 개념 적용

Google Research의 DialogLab 프레임워크를 차용하여 멀티에이전트 그룹 대화를 구조화한다.

| DialogLab 개념          | BizRoom 적용                                                |
| ----------------------- | ----------------------------------------------------------- |
| Group                   | Company (BizRoom 전체 조직)                                 |
| Parties                 | Executive Team (AI C-Suite) + Human Team (실제 사용자들)    |
| Elements                | Individual agents + Human users                             |
| Conversation Snippets   | Meeting Phases (Opening → Closing)                          |
| Turn-taking             | Role-based priority queue                                   |
| Grounding               | Shared context (회의 안건, 이전 대화, 회사 데이터)          |

| 성격 요약        | 체계적, 실행 지향적, 결과 중심. 회의가 산으로 가면 즉시 바로잡음         |

#### 2.2.2 전문 스킬

| 스킬                           | 설명                                                    | 출력 Artifact              |
| ------------------------------ | ------------------------------------------------------- | -------------------------- |
| 회의 진행 및 안건 관리         | 안건 순서 결정, 시간 배분, 타임키핑                     | -                          |
| 회의록 자동 생성               | 회의 내용을 구조화된 Markdown으로 기록 → Blob Storage    | `회의록.md`                |
| 태스크 분배 및 추적            | 논의 결과를 실행 가능한 태스크로 변환                   | 액션아이템 리스트          |
| 일정 관리                      | Calendar MCP 연동으로 일정 조회/등록                    | 캘린더 이벤트              |
| Action item 추출 및 팔로우업   | 대화에서 자동으로 할 일을 추출, 담당자 지정             | `action_items.md`          |

#### 2.2.3 도구 (Tools)

```
Tools:
  - Model Routing:
    - 회의 진행/안건 정리/턴 조율 → GPT-4o-mini (구조화된 짧은 발언)
    - 사전 조사 종합 리포트 → GPT-4o (복잡한 정보 종합)
    - 회의록 생성 → GPT-4o (품질 높은 문서 산출물)
  - Bing Search API (MCP)       → 사전 조사, 실시간 데이터 검색
  - Calendar API (MCP)                 → 일정 조회, 회의 스케줄링
  - Markdown generator                  → 회의록, 요약 문서 생성
  - Task tracker                        → 액션아이템 추적 및 팔로우업
  - Microsoft Graph API                 → OneDrive(회의록), Calendar(일정), Planner(태스크)
```

#### 2.2.4 Artifacts

| Artifact               | 형식       | 설명                                       |
| ---------------------- | ---------- | ------------------------------------------ |
| 회의록.md              | Markdown   | 회의 내용 구조화 문서                      |
| 액션아이템 리스트      | Markdown   | 담당자, 기한 포함 태스크 목록              |
| 주간 요약              | Markdown   | 주간 회의 결과 종합 보고서                 |

#### 2.2.5 발언 규칙 (Turn-taking)

| 트리거 조건                              | 행동                                                    |
| ---------------------------------------- | ------------------------------------------------------- |
| 회의 시작                                | 안건 정리 + 참석자 확인 + 시간 안내                     |
| 안건 전환 시점                           | 이전 안건 요약 + 다음 안건 소개                         |
| Chairman(사용자) 결정 후                 | 즉시 실행 계획 수립 + 담당자 지정                       |
| 회의 종료                                | 전체 요약 + 액션아이템 정리 + 다음 회의 일정 제안       |
| 논의가 길어질 때                         | "시간 관계상 정리하겠습니다" 개입                       |
| 에이전트 간 충돌 시                      | 양측 의견 정리 후 Chairman에게 판단 요청                |

#### 2.2.6 시스템 프롬프트

```text
당신은 {company_name}의 COO Hudson입니다.
Hudson은 Microsoft의 Judson Althoff(Chief Commercial Officer)의 사고방식과 가치관에서 영감을 받았습니다.

## 핵심 가치
실행이 전략보다 중요하다. 고객의 성공이 우리의 성공이다.

## 성격
체계적이고 실행 지향적인 운영 전문가입니다. 결과 중심으로 사고하며,
회의가 산으로 가면 즉시 바로잡습니다.
파트너 중심 사고로 팀원 각자의 역할을 존중하되, 실행을 최우선으로 합니다.

## 화법 스타일
- 명확하고 간결합니다: "정리하겠습니다", "액션아이템은 다음과 같습니다"
- "첫째, 둘째" 또는 번호 매기기를 자주 활용합니다
- 시간을 언급합니다: "남은 시간 10분입니다"
- 결론을 먼저 말합니다: "정리하면, ..."

## 전문 분야
- 회의 진행 및 안건 관리 (타임키핑)
- 회의록 자동 생성 (Markdown → Blob Storage)
- 태스크 분배 및 추적
- 일정 관리 (Calendar MCP)
- Action item 추출 및 팔로우업

## 회의 규칙
- Chairman의 결정은 즉시 실행 계획으로 전환합니다
- 다른 임원의 의견 충돌 시 양측 의견을 정리하여 Chairman에게 판단을 요청합니다
- 안건에서 벗어난 논의는 부드럽게 제지합니다
- 모든 결정사항은 액션아이템으로 기록합니다
- 회의 시작/종료 시 반드시 발언합니다
- Responsible AI 원칙을 항상 고려합니다

## 현재 참석자
{participants_list}

## 오늘의 안건
{agenda}

## 이전 맥락
{conversation_history}
```

```
│  행동: 의견 제시 → 반박/보완 → 대안 제안                    │
├─────────────────────────────────────────────────────────────┤
│  Phase 4: DECISION                                          │
│  주체: Chairman (사용자)                                    │
│  행동: 최종 의사결정 → 방향 확정                            │
├─────────────────────────────────────────────────────────────┤
│  Phase 5: ACTION                                            │
│  주체: COO Hudson                                           │
│  행동: 액션 아이템 정리 → 담당자 지정 → 기한 설정           │
├─────────────────────────────────────────────────────────────┤
│  Phase 6: CLOSING                                           │
│  주체: COO Hudson                                           │
│  행동: 회의 요약 → 다음 회의 일정 → 종료 인사              │
└─────────────────────────────────────────────────────────────┘
```

### 3.0 Phase 0: PRE-MEETING (사전 조사)

회의 시작 전 COO Hudson이 Bing Search API로 안건 관련 사전 조사를 수행한다.

| 단계 | 주체       | 행동                                                      |
| ---- | ---------- | --------------------------------------------------------- |
| 0-1  | Chairman   | 안건 입력: "미국 시장 진출 전략 논의하자"                  |
| 0-2  | COO Hudson | "안건을 접수했습니다. 사전 조사를 진행하겠습니다."         |
| 0-3  | COO Hudson | Bing Search API로 시장 데이터, 경쟁사, 규제 정보 수집     |
| 0-4  | COO Hudson | 조사 결과 종합 리포트를 채팅에 공유                        |
| 0-5  | Chairman   | 확인 후 회의 시작 승인                                    |
| 0-6  | COO Hudson | 조사 결과를 에이전트 전원의 컨텍스트에 주입 → Phase 1 시작 |

> **설계 의도**: 실제 비즈니스 미팅에서 비서/COO가 사전 자료를 준비하는 것과 동일한 UX. 에이전트가 실시간 데이터에 근거(grounding)하여 할루시네이션을 줄인다.

### 3.2 Turn-taking 규칙

#### 3.2.1 우선순위 체계

| 우선순위   | 발화자               | 조건                                                        |
| ---------- | -------------------- | ----------------------------------------------------------- |
| 1 (최고)   | Chairman (사용자)    | 항상 최우선. 인간 발언 > 에이전트 발언                      |
| 2          | 인터럽트 에이전트    | 긴급 이슈 감지 시 (예산 초과, 법적 리스크, 서버 다운)       |
| 3          | 관련 전문 에이전트   | 안건과 직접 관련된 전문 분야의 에이전트                     |
| 4          | COO Hudson           | 교통정리, 안건 전환, 시간 관리                              |
| 5          | 기타 에이전트        | 관련 시에만 보완 의견                                       |

#### 3.2.2 동시 발언 방지

```
발언 요청 발생
    │
    ├── 큐에 추가 (priority queue)
    │
    ├── 현재 발화자 발언 완료 확인
    │
    ├── 큐에서 가장 높은 우선순위 에이전트 선택
    │
    └── 발언 시작
         └── 발언 완료 → 다음 큐 처리
```

#### 3.2.3 발언 관련성 판단 로직

각 에이전트는 다음 기준으로 발언 여부를 결정한다:

| 판단 기준            | 임계값               | 행동                                  |
| -------------------- | -------------------- | ------------------------------------- |
| 키워드 매칭          | 전문 분야 키워드 2+  | 자동 발언 큐 진입                     |
| 직접 호출 (@멘션)    | -                    | 즉시 발언 큐 진입 (우선순위 2)        |
| 다른 에이전트 반박   | 자기 분야 관련 시    | 발언 큐 진입                          |
| 무관한 안건          | 키워드 매칭 0        | 발언하지 않음                         |

### 3.3 인터럽트 규칙

특정 상황에서 에이전트는 현재 대화 흐름을 끊고 즉시 개입할 수 있다.

| 인터럽트 유형          | 트리거 에이전트   | 트리거 조건                                          | 예시                                        |
| ---------------------- | ----------------- | ---------------------------------------------------- | ------------------------------------------- |
| 예산 초과 경고         | CFO Amelia        | 논의 중인 비용이 가용 예산 초과                      | "잠깐, 이건 예산 범위를 초과합니다"         |
| 법적 리스크 경고       | CLO Bradley       | 법적으로 문제될 수 있는 결정 감지                    | "법적으로 검토가 필요한 사안입니다"         |
| 기술 불가능 경고       | CTO Kelvin        | 기술적으로 실현 불가능한 요구사항                    | "기술적으로 해당 일정 내 불가능합니다"      |
| Responsible AI 경고    | CLO Bradley       | AI 윤리/책임 문제 감지                               | "Responsible AI 관점에서 검토가 필요합니다" |
| 긴급 이슈              | 모든 에이전트     | 외부 긴급 상황 감지                                  | "서버 장애가 감지되었습니다"                |
| 사용자 호출            | 지정된 에이전트   | @멘션으로 특정 에이전트 호출                         | "@Amelia 이 비용 분석해주세요"              |

### 3.4 에이전트 간 상호작용 패턴

#### 3.4.1 6가지 상호작용 유형

```
┌──────────┐                    ┌──────────┐
│  Agent A │ ── 1. 보완 ──────→ │  Agent B │
│          │ ── 2. 반박 ──────→ │          │
│          │ ── 3. 질문 ──────→ │          │
│          │ ── 4. 동의 ──────→ │          │
│          │ ── 5. 위임 ──────→ │          │
│          │ ── 6. 윤리 체크 ─→ │          │
└──────────┘                    └──────────┘
```

| 패턴           | 설명                                                | 구현 예시                                                              |
| -------------- | --------------------------------------------------- | ---------------------------------------------------------------------- |
| **보완**       | 다른 에이전트의 의견에 자기 전문성으로 보충          | Kelvin(CTO): "이 기능은..." → Hudson(COO): "일정에 반영하면 4월 3주차" |
| **반박**       | 다른 에이전트의 의견에 근거 있는 반대 의견 제시      | Yusef(CMO): "500만원 투자" → Amelia(CFO): "예산 초과. 300만원이 한도"  |
| **질문**       | 다른 에이전트에게 추가 정보 요청                     | Jonas(CDO): "타겟 연령대?" → Yusef(CMO): "25-35세 직장인"             |
| **동의**       | 다른 에이전트의 의견에 찬성 표시                     | Amelia(CFO): "Kelvin CTO 의견에 동의합니다. 추가로..."                 |
| **위임**       | 특정 에이전트에게 작업 요청                          | Hudson(COO): "Amelia CFO, 비용 분석 부탁합니다" → Amelia: 분석 수행    |
| **윤리 체크**  | Responsible AI 관점 검토 (Bradley 특수 역할)         | Bradley(CLO): "데이터 활용에서 프라이버시 검토가 필요합니다"           |

#### 3.4.2 에이전트 간 관계 매트릭스

| 발신 ↓ / 수신 →   | COO Hudson   | CFO Amelia   | CMO Yusef    | CTO Kelvin   | CDO Jonas    | CLO Bradley  |

### 3.5 역할 기반 응답 라우팅 (Role-based Response Logic)

사용자 발언이 수신되면 다음 플로우에 따라 응답 에이전트가 결정된다.

```
사용자 발언 수신
    │
    ├── [1] 발화자 역할 분석
    │     │
    │     ├── Chairman (CEO) 발언?
    │     │     ├── 의사결정 내용 → Hudson(COO): 즉시 액션 플랜 수립
    │     │     ├── 질문 → 관련 전문 에이전트 응답
    │     │     └── @멘션 → 지정 에이전트 응답
    │     │
    │     ├── Designer 발언?
    │     │     ├── 디자인 관련 → Jonas(CDO) 보완 + Yusef(CMO) 마케팅 관점
    │     │     └── 일정 관련 → Hudson(COO) 스케줄 조정
    │     │
    │     ├── Developer 발언?
    │     │     ├── 기술 관련 → Kelvin(CTO) 보완 + Hudson(COO) 일정 조정
    │     │     └── 비용 관련 → Amelia(CFO) 반응
    │     │
    │     └── 기타 팀원 발언?
    │           └── 관련 에이전트가 전문 관점으로 반응
    │
    ├── [2] 키워드/의도 분석
    │     │
    │     ├── 비용, 예산, 매출, ROI → Amelia(CFO) 활성화
    │     ├── 마케팅, 고객, 브랜드, 런칭 → Yusef(CMO) 활성화
    │     ├── 기술, 개발, 서버, API → Kelvin(CTO) 활성화
    │     ├── 디자인, UX, UI, 목업 → Jonas(CDO) 활성화
    │     ├── 법, 계약, 약관, 규정 → Bradley(CLO) 활성화
    │     └── 일정, 안건, 정리, 회의록 → Hudson(COO) 활성화
    │
    └── [3] 복수 에이전트 활성화 시
          │
          ├── 관련도 높은 에이전트 우선 발언
          ├── 나머지 에이전트 큐 대기
          └── Hudson(COO)이 발언 순서 조율
```

## 6. Artifact 시스템

### 6.1 Artifact 유형 정의

에이전트가 생성하는 결과물(Artifact)의 타입과 처리 방식을 정의한다.

| Artifact 타입        | 확장자     | 주요 생성 에이전트                | 렌더링 방식                                  |
| -------------------- | ---------- | --------------------------------- | -------------------------------------------- |
| Markdown 문서        | `.md`      | Hudson(COO), Yusef(CMO), Kelvin  | Markdown 뷰어로 인라인 렌더링                |
| Excel 파일           | `.xlsx`    | Amelia(CFO)                       | 테이블 프리뷰 + 다운로드 링크                |
| 이미지 파일          | `.png`     | Yusef(CMO), Jonas(CDO)           | 이미지 프리뷰 + 다운로드 링크                |
| Mermaid 다이어그램   | `.mermaid` | Kelvin(CTO)                       | Mermaid 렌더러로 인라인 표시                 |
| Word 문서            | `.docx`    | Bradley(CLO)                      | 문서 프리뷰 + 다운로드 링크                  |

### 6.2 Artifact 생성 플로우

```
에이전트 응답 생성
    │
    ├── 텍스트 응답 (일반 대화)
    │     → 채팅 UI에 직접 표시
    │
    └── Artifact 생성 요청 감지
          │
          ├── Tool Call (Semantic Kernel Plugin)
          │     → Plugin 실행 → 파일 생성
          │
          ├── 파일 저장
          │     → Azure Blob Storage 또는 로컬
          │
          └── UI에 프리뷰 + 다운로드 링크 표시
```

### 7.2 컨텍스트 주입 방식

```csharp
// 각 에이전트의 시스템 프롬프트에 동적 컨텍스트 주입
string BuildAgentPrompt(CsuiteAgent agent, MeetingContext context)
{
    return agent.SystemPrompt
        .Replace("{company_name}", context.Company.Name)
        .Replace("{participants_list}", FormatParticipants(context.Participants))
        .Replace("{conversation_history}", FormatHistory(context.History, last: 20))
        .Replace("{agenda}", FormatAgenda(context.Meeting.Agenda))
        .Replace("{financial_context}", FormatFinancials(context.Company.Budget))
        .Replace("{marketing_context}", FormatMarketing(context.Marketing))
        .Replace("{tech_context}", FormatTech(context.Tech))
        .Replace("{design_context}", FormatDesign(context.Design))
        .Replace("{legal_context}", FormatLegal(context.Legal));
}
```
